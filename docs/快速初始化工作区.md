# 快速初始化工作区

这份文档用于从零准备一个可运行的 SpecLane 工作区。当前标准结构是：一个主 skill、一个 `workspace.yml`、多个需求实例，每个需求按阶段保存自己的输入、规格、研发交付、测试和归档产物。

## 1. 推荐目录

```text
ai-workspace/
├── workspace.yml
├── docs/
│   ├── 需求分析与实现指南.md
│   └── 开发规范.md
├── .speclane/                         # 运行态，用户一般不编辑
└── demands/
    └── 10-your-demand/
        ├── input/
        │   ├── 需求.md
        │   └── references/
        ├── pm/
        ├── spec/
        │   ├── openspec/
        │   │   ├── changes/
        │   │   └── specs/
        │   └── bridge/
        │       └── todo.md
        ├── rd/
        │   └── output/
        ├── qa/
        └── archive/

code/
├── frontend-app/
├── item-core/
└── other-service/
```

约定：

- `workspace.yml` 是唯一用户可维护的工作区配置。
- `.speclane/` 只保存状态、session、锁等运行态。
- `demands/<demand-name>/input/` 保存原始需求和参考资料。
- `demands/<demand-name>/spec/` 保存 OpenSpec change 和桥接 todo。
- `demands/<demand-name>/rd/output/` 保存研发交付报告。
- `pm/`、`qa/`、`archive/` 是后续产品、测试和归档扩展预留目录。

## 2. 安装

```bash
npm install -g @gary-coding/speclane@latest
speclane init
```

`speclane init` 是交互式向导，会完成：

1. 检查 Python、Node、OpenSpec CLI。
2. 选择安装到 Codex、Claude 或两者。
3. 选择业务工作区。
4. 配置代码目录和首个需求名。
5. 选择 `openspec` 或 `todo` 模式。
6. 生成 `workspace.yml`、需求阶段目录和 `/sl:*` 快捷命令。

非交互初始化示例：

```bash
speclane init \
  --yes \
  --install both \
  --workspace /path/to/ai-workspace \
  --code-path ../code \
  --demand-name 10-your-demand \
  --source openspec \
  --mode auto
```

同步最新 skill：

```bash
speclane sync --target both
```

安装或修复快捷命令：

```bash
speclane commands install --workspace /path/to/ai-workspace --target all
```

诊断工作区：

```bash
speclane doctor --workspace /path/to/ai-workspace
```

## 3. workspace.yml

推荐使用 `demands[]` 管理多个需求。路径里统一使用 `${demand_name}`，这样新增需求时只需要写一次 `name`。

OpenSpec + auto 示例：

```yaml
version: 1
mode: auto
workflow_source: openspec
code_path: ../code
reference_files:
  - docs/需求分析与实现指南.md
  - docs/开发规范.md
openspec:
  changes_dir: demands/${demand_name}/spec/openspec/changes

demands:
  - name: 10-your-demand
    desc: 示例需求描述
    workflow_source: openspec
    mode: auto
    demand_file: demands/${demand_name}/input/需求.md
    todo_file: demands/${demand_name}/spec/bridge/todo.md
    output_dir: demands/${demand_name}/rd/output
    reference_files:
      - docs/需求分析与实现指南.md
      - docs/开发规范.md
    code_path: ../code
    openspec:
      changes_dir: demands/${demand_name}/spec/openspec/changes
```

todo + auto 示例：

```yaml
version: 1
mode: auto
workflow_source: todo
code_path: ../code
reference_files: []

demands:
  - name: 10-your-demand
    desc: 示例需求描述
    workflow_source: todo
    mode: auto
    todo_file: demands/${demand_name}/spec/bridge/todo.md
    output_dir: demands/${demand_name}/rd/output
    reference_files: []
    code_path: ../code
```

## 4. OpenSpec 初始化

SpecLane 的 OpenSpec change 默认写入需求目录：

```text
demands/<demand-name>/spec/openspec/changes/<change-name>/
```

如果你需要使用 OpenSpec CLI 的官方命令或项目配置，可以在工作区根目录执行：

```bash
openspec init . --tools codex,claude
```

注意：不要在 `openspec/` 子目录里执行初始化，也不要把 SpecLane 的 `changes_dir` 指向 `openspec/openspec/changes` 这类嵌套路径。

## 5. 开始一个需求

准备需求文件：

```text
demands/10-your-demand/input/需求.md
```

OpenSpec 模式按顺序给 AI 发送：

```text
/sl:propose your-change-name
请根据当前 workspace 的 demand_file 生成或完善 OpenSpec change。
不要改代码。
默认使用中文回复。
```

```text
/sl:bridge
针对当前 OpenSpec change 生成桥接 todo，并总结待审核项。
不要改代码，不要进入实现。
默认使用中文回复。
```

人工审核：

```text
demands/10-your-demand/spec/bridge/todo.md
```

审核通过后：

```text
/sl:apply
我已审核当前桥接 todo，可以进入交付阶段。
如果没有硬阻塞，自动推进到实现、自查、审查和验证。
默认使用中文回复。
```

todo 模式可以直接准备：

```text
demands/10-your-demand/spec/bridge/todo.md
```

然后发送：

```text
/sl:apply
请根据当前 workspace 的 todo_file 推进交付工作流。
默认使用中文回复。
```

## 6. 多需求

新增需求：

```text
/sl:demand new 11-another-demand 新需求描述
```

切换当前需求：

```text
/sl:demand use 11-another-demand
```

查看需求列表：

```text
/sl:demand list
```

也可以在主命令后显式指定需求：

```text
/sl:propose change-name --demand 11-another-demand
/sl:bridge --demand 11-another-demand
/sl:apply --demand 11-another-demand
```

## 7. 常见问题

### `/sl:propose` 会自动生成 todo 吗？

不会。`/sl:propose` 只生成或完善 OpenSpec change。只有显式 `/sl:bridge` 才能生成或覆盖 `todo_file`。

### `workspace.yml` 可以让 AI 自动改吗？

不可以。`workspace.yml` 是用户维护的工作区契约，AI 只能读取和校验。

### 审核 todo 后发现需求不对怎么办？

先修改 `input/需求.md` 或飞书文档，再重新执行 `/sl:propose <change-name>`，然后重新 `/sl:bridge`。

### 交付报告在哪里？

```text
demands/<demand-name>/rd/output/<session-id>/
```

### 运行态在哪里？

```text
.speclane/demands/<demand-name>/
```

这里保存的是机器状态，不是需求配置。
