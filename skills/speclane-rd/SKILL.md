# SpecLane RD

Use this skill when the user sends `/sl:*` or asks to run the speclane workflow.

## Language

- Reply in the same language as the user's latest message by default.
- If the user explicitly asks for Chinese or English, follow that language.
- Keep command names, file names, JSON keys, and CLI output keys unchanged.
- Reports may contain mixed technical English such as OpenSpec, Skill, todo.md, verify, and archive.

## Command Routing

`/sl:*` is a workflow instruction for AI. It is not a shell command and must not be mapped to OpenSpec `/opsx:*`.

Supported commands:

- `/sl:init`
- `/sl:propose <change-name>`
- `/sl:bridge`
- `/sl:plan`
- `/sl:apply`
- `/sl:review`
- `/sl:verify`
- `/sl:archive-check`
- `/sl:archive`
- `/sl:status`

For command details, read [references/commands/common.md](references/commands/common.md) plus only the matching command file:

## Minimal Required Steps

1. Read `<workspace>/workspace.yml`.
2. Identify `workflow_source`, `mode`, `todo_file`, `demand_file`, `reference_files`, `code_path`, `output_dir`, and optional `openspec` fields.
3. Preflight with `python3 scripts/run-workflow.py route-check --command-text "<command>"` when possible.
4. Route the current command through `python3 scripts/run-workflow.py route-sl --command-text "<command>"` when preflight passes.
5. Obey script state validation and script reply constraints.
6. Report compactly: result, blockers, allowed next command, and key artifact paths.

## Hard Constraints

- Never edit `<workspace>/workspace.yml`; it is a user-maintained contract.
- Never manually create, edit, or fake `.speclane/current-session.json`, `.speclane/sl-state.json`, `.speclane/sessions/**/status.json`, `plan.json`, `review.json`, `verify.json`, `notification.json`, or standard output reports.
- Standard JSON and Markdown artifacts must be generated by workflow scripts.
- Execute only the `/sl:*` command explicitly requested by the current user message. Suggested next commands are text only.
- If a script prints `final_reply_must` or `sl_reply_constraint_begin` / `sl_reply_constraint_end`, the final reply must obey it.
- `auto` only applies inside `/sl:apply`; `/sl:init`, `/sl:propose`, `/sl:bridge`, and `/sl:plan` stop after their own stage.
- In `openspec` mode, `/sl:propose <change-name>` requires an explicit change name. Do not infer it from demand title or `demand_name`.
- In `openspec` mode, only explicit `/sl:bridge` may generate or overwrite `todo_file`.
- `/sl:bridge` output must be reviewed by the user before `/sl:apply`.
- `/sl:propose` must not call bridge or modify `todo_file`; its next allowed command is `/sl:bridge`.
- `/sl:bridge` must not call plan/apply/review/verify or modify code.
- `/sl:verify` must pass before suggesting `/sl:archive-check`.
- `/sl:archive-check` must report `safe_merge` before suggesting `/sl:archive`.
- Notifications must be sent only by `python3 scripts/run-workflow.py verify`; AI must not call Feishu/PushPlus webhooks directly.
- `notification.json` is the only notification success evidence.

## Context Discipline

- Prefer compact summaries and artifact paths over pasting full files into replies.
- `reference_files` are strong context, but scripts summarize large files by default; read full files only when necessary for the current command.
- In OpenSpec mode, avoid rereading `proposal.md`, `design.md`, and `specs/**/*.md` unless the current stage needs them.
- Prefer `plan-summary.json` for downstream context; use `plan.json` only when detailed planning data is required.
- Prefer `discovery-summary.json` over `discovery.json`; read full discovery only for detailed code evidence.
- Prefer `route-sl --json` when a machine-readable execution summary is useful.
- Final replies should be compact. Detailed reports belong in `output_dir/<session_id>/`.

## State Summary

OpenSpec mode state is script-managed in `<workspace>/.speclane/sl-state.json`:

- after `/sl:propose`: only `/sl:bridge`
- after `/sl:bridge`: `/sl:apply` or `/sl:plan`
- after `/sl:plan`: only `/sl:apply`
- after successful `/sl:verify`: only `/sl:archive-check`
- after safe `/sl:archive-check`: only `/sl:archive`

Todo mode uses current session `status.json` and `todo-state.json`; do not write OpenSpec `sl-state.json` for todo mode.

## Script Entry Points

- Preflight router: `python3 scripts/run-workflow.py route-check --command-text "<command>"`
- Main router: `python3 scripts/run-workflow.py route-sl --command-text "<command>"`
- JSON router summary: `python3 scripts/run-workflow.py route-sl --command-text "<command>" --json`
- Status: `python3 scripts/run-workflow.py status`
- Plan: `python3 scripts/run-workflow.py plan`
- Review: `python3 scripts/run-workflow.py review`
- Verify: `python3 scripts/run-workflow.py verify`

OpenSpec-specific execution details are enforced by the command files under [references/commands/](references/commands/).
