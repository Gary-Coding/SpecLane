# SpecTrace RD

Use this skill when the user sends `/st:*` or asks to run the spectrace workflow.

## Command Routing

`/st:*` is a workflow instruction for AI. It is not a shell command and must not be mapped to OpenSpec `/opsx:*`.

Supported commands:

- `/st:init`
- `/st:propose <change-name>`
- `/st:bridge`
- `/st:plan`
- `/st:apply`
- `/st:review`
- `/st:verify`
- `/st:archive-check`
- `/st:archive`
- `/st:status`

For command details, read [references/commands/common.md](references/commands/common.md) plus only the matching command file:

- `/st:propose`: [references/commands/propose.md](references/commands/propose.md)
- `/st:bridge`: [references/commands/bridge.md](references/commands/bridge.md)
- `/st:plan`: [references/commands/plan.md](references/commands/plan.md)
- `/st:apply`: [references/commands/apply.md](references/commands/apply.md)
- `/st:review`: [references/commands/review.md](references/commands/review.md)

## Minimal Required Steps

1. Read `<workspace>/workspace.yml`.
2. Identify `workflow_source`, `mode`, `todo_file`, `demand_file`, `reference_files`, `code_path`, `output_dir`, and optional `openspec` fields.
3. Preflight with `python3 scripts/run-workflow.py route-check --command-text "<command>"` when possible.
4. Route the current command through `python3 scripts/run-workflow.py route-st --command-text "<command>"` when preflight passes.
5. Obey script state validation and script reply constraints.
6. Report compactly: result, blockers, allowed next command, and key artifact paths.

## Hard Constraints

- Never edit `<workspace>/workspace.yml`; it is a user-maintained contract.
- Never manually create, edit, or fake `.spectrace/current-session.json`, `.spectrace/se-state.json`, `.spectrace/sessions/**/status.json`, `plan.json`, `review.json`, `verify.json`, `notification.json`, or standard output reports.
- Standard JSON and Markdown artifacts must be generated by workflow scripts.
- Execute only the `/st:*` command explicitly requested by the current user message. Suggested next commands are text only.
- If a script prints `final_reply_must` or `se_reply_constraint_begin` / `se_reply_constraint_end`, the final reply must obey it.
- `auto` only applies inside `/st:apply`; `/st:init`, `/st:propose`, `/st:bridge`, and `/st:plan` stop after their own stage.
- In `openspec` mode, `/st:propose <change-name>` requires an explicit change name. Do not infer it from demand title or `demand_name`.
- In `openspec` mode, only explicit `/st:bridge` may generate or overwrite `todo_file`.
- `/st:bridge` output must be reviewed by the user before `/st:apply`.
- `/st:propose` must not call bridge or modify `todo_file`; its next allowed command is `/st:bridge`.
- `/st:bridge` must not call plan/apply/review/verify or modify code.
- `/st:verify` must pass before suggesting `/st:archive-check`.
- `/st:archive-check` must report `safe_merge` before suggesting `/st:archive`.
- Notifications must be sent only by `python3 scripts/run-workflow.py verify`; AI must not call Feishu/PushPlus webhooks directly.
- `notification.json` is the only notification success evidence.

## Context Discipline

- Prefer compact summaries and artifact paths over pasting full files into replies.
- `reference_files` are strong context, but scripts summarize large files by default; read full files only when necessary for the current command.
- In OpenSpec mode, avoid rereading `proposal.md`, `design.md`, and `specs/**/*.md` unless the current stage needs them.
- Prefer `plan-summary.json` for downstream context; use `plan.json` only when detailed planning data is required.
- Prefer `discovery-summary.json` over `discovery.json`; read full discovery only for detailed code evidence.
- Prefer `route-st --json` when a machine-readable execution summary is useful.
- Final replies should be compact. Detailed reports belong in `output_dir/<session_id>/`.

## State Summary

OpenSpec mode state is script-managed in `<workspace>/.spectrace/se-state.json`:

- after `/st:propose`: only `/st:bridge`
- after `/st:bridge`: `/st:apply` or `/st:plan`
- after `/st:plan`: only `/st:apply`
- after successful `/st:verify`: only `/st:archive-check`
- after safe `/st:archive-check`: only `/st:archive`

Todo mode uses current session `status.json` and `todo-state.json`; do not write OpenSpec `se-state.json` for todo mode.

## Script Entry Points

- Preflight router: `python3 scripts/run-workflow.py route-check --command-text "<command>"`
- Main router: `python3 scripts/run-workflow.py route-st --command-text "<command>"`
- JSON router summary: `python3 scripts/run-workflow.py route-st --command-text "<command>" --json`
- Status: `python3 scripts/run-workflow.py status`
- Plan: `python3 scripts/run-workflow.py plan`
- Review: `python3 scripts/run-workflow.py review`
- Verify: `python3 scripts/run-workflow.py verify`

OpenSpec-specific execution details are enforced by the command files under [references/commands/](references/commands/).
